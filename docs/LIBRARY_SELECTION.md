# ライブラリ選定

## SQS クライアント

### 背景

2026年3月現在、Python には Spring の `@SqsListener` のような SQS Pub/Sub 処理に特化した代表的なフレームワークがなく、いくつかのライブラリが存在する程度です。
そのため、AWS SDK である boto3 を直接使用し、SQS の送受信処理をラップする実装を行っています。

### 選定

| ライブラリ                                          | 概要                                  | 採否  |
|------------------------------------------------|-------------------------------------|-----|
| [boto3](https://pypi.org/project/boto3/)       | AWS 公式 SDK。SQS の全操作をサポート            | 採用  |
| [aioboto3](https://pypi.org/project/aioboto3/) | boto3 の非同期ラッパー。asyncio ベースで非同期処理が可能 | 不採用 |

### 理由

- 今回のシナリオでは、Subscriber はイベント受信後に同期的な処理（Redis への保存）を行う前提のため、非同期処理の必要性がない
- boto3 は AWS 公式かつ最も広く使われており、情報量・安定性ともに十分
- 将来的にイベント処理を非同期で行う要件が出た場合は aioboto3 への移行を検討する
