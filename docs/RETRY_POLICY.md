# リトライポリシー

## Subscriber の動作イメージ

Subscriber は常駐プロセスとして動作し、SQS キューを継続的にポーリングします。
Spring の `@SqsListener` と同様に、キューが利用できない状態でもプロセスは終了せず、復旧を待ち続けます。

### キュー取得 (`_get_queue_url`)

- キューが存在しない場合、一定間隔で無限リトライする
- リトライ間隔は環境変数 `SQS_RETRY_INTERVAL_SECONDS` で設定可能（デフォルト: 2 秒）
- `compose.yaml` の subscriber サービスの `environment` で上書きできる
  ```yaml
  subscriber:
    environment:
      - SQS_RETRY_INTERVAL_SECONDS=5
  ```
- Publisher によるキュー作成待ちや、SQS の一時的な障害に対応するための設計
- リトライ中はログを出力し、状況を把握できるようにする

### メッセージ受信 (`receive`)

- ロングポーリング（`WaitTimeSeconds=20`）でメッセージを待機する
- メッセージがなければ空リストを返し、次のポーリングに移る
- メインループが `while True` で回り続けるため、常にキューを監視する

### メッセージ処理・削除

- メッセージ受信後、Redis への保存が完了してから SQS のメッセージを削除する
- 保存前にプロセスが落ちた場合、メッセージは SQS の可視性タイムアウト後に再配信される

### 全体のフロー

```
起動
  └─ キュー取得を試みる
       ├─ 成功 → ポーリング開始
       └─ 失敗 → 待機（SQS_RETRY_INTERVAL_SECONDS） → 再試行（無限リトライ）

ポーリング中
  └─ メッセージ受信（ロングポーリング 20秒）
       ├─ メッセージあり → Redis に保存 → SQS から削除 → 次のポーリングへ
       └─ メッセージなし → 次のポーリングへ
```

## 重複処理の防止

標準 SQS キューは at-least-once 配信のため、同一メッセージが複数回配信される可能性があります。
本サンプルでは Subscriber を 1 コンテナで運用することを前提とし、重複処理のリスクを最小限に抑えています。

## スケーリングが必要な場合

Subscriber のスケールアウトが必要になった場合は、以下を検討してください。

- FIFO キュー — メッセージの順序保証と exactly-once 処理が必要な場合
- SNS + SQS ファンアウト — 複数の Subscriber に同一メッセージを配信する場合
